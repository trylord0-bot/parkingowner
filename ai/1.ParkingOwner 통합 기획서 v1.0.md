## 3. 사용자 역할 체계

### 3-1. 역할 정의

| 역할 | 수 | 주요 권한 |
|------|-----|-----------|
| **앱관리자** | 개발자 1명 | 시스템 전체 관리, 이의신청 처리 |
| **단지관리자** | 1명 / 단지 | 단지 생성, 관리인 임명, 모든 기능 접근 |
| **주차관리인** | N명 / 단지 | 차량 전체 관리, 세대원 승인, 실시간 스캔 |
| **세대원** | N명 / 단지 | 차량 조회/스캔(조회 전용), 본인 세대 차량 등록 요청 |

### 3-2. 역할 관계

```
앱관리자 (개발자)
    └── 단지관리자 (1명/단지)
            ├── 주차관리인 (N명/단지)  ← 단지관리자 겸직 가능
            └── 세대원 (N명/단지)      ← 주차관리인 겸직 가능
```

> - 단지관리자는 주차관리인 역할 겸직 가능
> - 주차관리인은 세대원이거나 외부 고용인 가능

## 4. 기술 스택

### 4-1. 앱 (Frontend)

| 항목 | 기술 | 비고 |
|------|------|------|
| 프레임워크 | **Flutter** | iOS/Android 동시 개발 |

| OCR 인식 | **PaddleOCR Lite** | 텍스트 인식 (온디바이스, 무료) |
| 위치기반 | **Geolocator** | GPS 기반 단지 리스트업 |
| 로컬 큐 | **SQLite (sqflite)** | OCR 수정 데이터 오프라인 큐 |

### 4-2. 백엔드 (Backend)

| 항목 | 기술 | 비고 |
|------|------|------|
| 런타임 | **Node.js** | |

| DB | **MariaDB** | 무료, MySQL 호환 |
| ORM | **Prisma** | 타입 안전, 마이그레이션 편리 |
| 인증 | **JWT + Refresh Token** | Stateless 표준 인증 |

| 프록시/SSL | **Nginx + Let's Encrypt** | 무료 SSL 자동갱신 |

### 4-3. OCR 파이프라인

```
카메라 프레임
    │
    ▼
[RF-DETR / ONNX Runtime]    ← 번호판 영역 탐지 (Bounding Box)
    │
    ▼
[이미지 크롭 + 전처리]
    │
    ▼
[PaddleOCR Lite]             ← 텍스트 인식 (한국/글로벌 번호판)
    │
    ▼
[규칙 기반 정규식 보정]       ← O↔0, I↔1, B↔8 등 오인식 교정
    │
    ▼
번호판 텍스트 확정 → 수동 수정 UI 제공
```

### 4-4. 한국 번호판 정규식 패턴

```
신형 (2019~): \d{2,3}[가-힣]\d{4}
구형:         [가-힣]{2}\d{2}[가-힣]\d{4}
```

## 5. 디자인 시스템

### 5-1. 브랜딩 컬러 — Steel Azure

파킹오너의 공식 브랜딩 테마는 **Steel Azure**입니다.
고급스럽고 신뢰감 있는 딥 블루 그레이 계열을 사용합니다.

#### Light Mode

```css
:root {
  --color-primary:    #1B3D6F;  /* Steel Navy   — 버튼, 헤더, 주요 액션 */
  --color-secondary:  #2D6098;  /* Ocean Blue   — 보조 버튼, 아이콘 */
  --color-accent:     #4A90C8;  /* Azure        — 링크, 하이라이트, 포인트 */
  --color-bg:         #F0F4F9;  /* Frost White  — 앱 전체 배경 */
  --color-surface:    #FFFFFF;  /* Pure White   — 카드, 모달, 시트 */
  --color-text:       #162035;  /* Deep Ink     — 본문 텍스트 */
  --color-subtext:    #607080;  /* Slate        — 보조 텍스트, 플레이스홀더 */
}
```

#### Dark Mode

```css
[data-theme="dark"] {
  --color-primary:    #4A90C8;  /* Azure        — 버튼, 헤더, 주요 액션 */
  --color-secondary:  #2D6098;  /* Ocean Blue   — 보조 버튼, 아이콘 */
  --color-accent:     #78B8E8;  /* Sky Steel    — 링크, 하이라이트, 포인트 */
  --color-bg:         #0C1520;  /* Abyss        — 앱 전체 배경 */
  --color-surface:    #142030;  /* Deep Steel   — 카드, 모달, 시트 */
  --color-text:       #D8E8F5;  /* Ice White    — 본문 텍스트 */
  --color-subtext:    #7890A8;  /* Steel Gray   — 보조 텍스트, 플레이스홀더 */
}
```

#### Semantic Colors (상태 구분 — 라이트/다크 공통 고정값)

```css
:root {
  --color-registered:    #2ECC71;  /* Emerald Green — 등록 차량 */
  --color-visitor:       #F1C40F;  /* Amber Yellow  — 방문 차량 */
  --color-unregistered:  #E74C3C;  /* Alert Red     — 미등록 차량 */
}
```

> ⚠️ Semantic Colors는 어떤 상황에서도 변경하지 않습니다.
> 카메라 스캔 화면에서 0.1초 안에 차량 상태를 식별해야 하므로,
> 이 세 가지 색상은 브랜딩 컬러와 혼용하거나 대체하지 않습니다.

#### 컬러 사용 규칙

| 규칙 | 내용 |
|------|------|
| Primary | 앱 내 가장 중요한 단 하나의 CTA 버튼에만 사용 |
| Semantic Color | 버튼 배경·일반 UI 요소 사용 금지. 텍스트/아이콘/레이블과 병행 사용 |
| Primary 중복 | 두 개 이상의 CTA에 동시 사용 금지 |
| 다크 모드 | Light Mode 토큰 그대로 사용 금지 |
| 흰색 텍스트 | Light Mode에서 `#FFFFFF` 텍스트 사용 금지 |

#### 접근성 기준 (WCAG 2.1 AA)

- 본문 텍스트 대비율: **최소 4.5:1 이상**
- 대형 텍스트(18px Bold 이상) 대비율: **최소 3:1 이상**
- Semantic Color는 색상만으로 구분하지 않고 **아이콘 또는 텍스트 레이블 병행** 표시

#### Type Scale

| Role | Size | Weight | Line Height | Letter Spacing | 용도 |
|------|:----:|:------:|:-----------:|:--------------:|------|
| **Display** | 32px | 700 | 1.2 | -0.5px | 메인 타이틀, 숫자 대시보드 |
| **Heading 1** | 24px | 700 | 1.3 | -0.3px | 섹션 제목, 모달 타이틀 |
| **Heading 2** | 18px | 700 | 1.4 | -0.2px | 카드 제목, 리스트 헤더 |
| **Body** | 15px | 400 | 1.6 | 0px | 일반 본문, 설명 텍스트 |
| **Caption** | 12px | 300 | 1.5 | 0.2px | 타임스탬프, 보조 정보, 레이블 |
| **Micro** | 10px | 400 | 1.4 | 0.2px | 뱃지, 탭바 레이블, 상태 태그 등 UI 최소 단위 |

> ⚠️ **최소 폰트 정책**
> - 일반 본문/설명 최소값: **12px (Caption)**
> - UI 컴포넌트 최소값: **10px (Micro)** — 뱃지, 탭바 레이블, 상태 태그 등 공간 제약이 있는 요소에 한해 허용
> - 9px 이하 절대 금지

#### 번호판 전용 스타일

  font-size: 28px;

| Weight | 수치 | 용도 |
|--------|:---:|------|

## 6. 단지(Complex) 관리

### 6-1. 단지 검색 및 선택 플로우

```
앱 실행 (로그인 후 첫 화면)
    │
    ▼
내 단지 목록 표시 + [단지 추가] 버튼
    │
    ▼
단지 검색 방식 선택
    ├── 🔍 주소 검색
    └── 📍 내 주변 (GPS 기반 반경 리스트업)
    │
    ▼
단지 선택
    ├── 이미 등록된 단지
    │       ├── 세대원: 등록 요청 or 초대코드 입력
    │       └── 관리인 주장 충돌 시: 이의신청 (앱관리자 이메일)
    └── 없는 단지
            └── 단지관리자로 직접 등록
```

### 6-2. 세대원 가입 방법

| 방법 | 설명 |
|------|------|
| **직접 요청** | 세대원이 단지 검색 후 등록 요청 → 주차관리인/단지관리자 승인 |
| **초대코드** | 주차관리인이 초대코드 발급 → 세대원이 코드 입력 → 즉시 승인 |

### 6-3. 관리인 이의신청

- 이미 등록된 단지에 관리인 충돌 발생 시
- 앱 내 이의신청 버튼 → 앱관리자(개발자)에게 이메일 자동 작성
- 앱관리자가 수동 검토 후 처리

## 8. 차량 관리

### 8-1. 차량 유형별 정책

| 구분 | 등록차량 | 방문차량 |
|------|----------|----------|
| 등록 주체 | 세대원 요청 → 관리인 승인 | 관리인 직접 or 세대원 요청 |
| 유효기간 | 없음 (영구) | 일/시/분 단위, 관리인 직접 지정 |
| 최대 기간 | — | 관리인이 자유 지정 |
| 대수 제한 | 없음 (표시만) | 없음 (표시만) |
| 스캔 표시 | ✅ 녹색 (`#2ECC71`) | 🟡 노랑 (`#F1C40F`) / 만료 시 🔴 빨강 |
| 미등록 | 🔴 빨강 (`#E74C3C`) | — |

### 8-2. 방문차량 유효기간 설정

- 단위: 일(Day) / 시(Hour) / 분(Minute) 조합
- 최대 기간: 주차관리인이 직접 지정 (제한 없음)
- 만료 임박 알림: 주차관리인 + 단지관리자에게 푸시 알림

## 12. OCR 수정 데이터 수집 (ML 학습 파이프라인)

### 12-1. 개념

> 사용자의 수동 수정 행위 = 가장 신뢰도 높은 Ground Truth 레이블

### 12-2. 전송 조건

| 조건 | 전송 여부 |
|------|----------|
| OCR 인식 후 수정 없이 확인 | ❌ 전송 안 함 |
| OCR 인식 후 수정하고 확인 | ✅ 전송 |
| OCR 인식 실패 + 수동 입력 | ✅ 전송 (이미지 + 정답 레이블) |
| 스캔 없이 수동 입력 전용 | ❌ 전송 안 함 |

### 12-3. 전송 데이터 구조

```json
{
  "type": "ocr_correction",
  "device_info": {
    "platform": "android | ios",
    "model": "기기 모델명",
    "os_version": "OS 버전"
  },
  "capture": {
    "image_original_url": "MinIO URL (전체 이미지)",
    "image_cropped_url": "MinIO URL (번호판 크롭)",
    "captured_at": "ISO8601"
  },
  "ocr_result": {
    "raw_text": "12가 3456",
    "confidence_score": 0.87,
    "model_version": "paddleocr-lite-v1.2"
  },
  "correction": {
    "corrected_text": "12나 3456",
    "corrected_by": "user_id",
    "corrected_at": "ISO8601"
  },
  "complex_id": "complex_id"
}
```

### 12-4. UX 정책

- 백그라운드 비동기 전송 (사용자 흐름 방해 없음)
- 전송 실패 시 로컬 SQLite 큐 저장 → 네트워크 복구 후 자동 재전송 (최대 3회)
- 3회 실패 시 조용히 폐기
- 크롭 이미지 우선 전송 (개인정보 보호)
- GPS 위치 미수집 (complex_id로 대체)

### 12-5. ML 학습 파이프라인

```
사용자 수정 데이터 누적 (OcrCorrectionLog)
    │
    ▼
데이터 전처리
    ├── 중복 제거
    ├── 번호판 패턴 유효성 검증 (정규식)
    ├── 이미지 품질 필터링 (블러/해상도)
    └── (선택) 관리자 수동 검수
    │
    ▼
학습 데이터셋 구성 (이미지 + 정답 레이블)
    │
    ▼
PaddleOCR Fine-tuning / RF-DETR 탐지 개선
    │
    ▼
모델 평가 (구버전 vs 신버전 정확도 비교)
    │
    ▼
앱 업데이트로 온디바이스 모델 교체
    │
    ▼
정확도 향상 → 수정 빈도 감소 (선순환)
```
